版本管理规范

作者 : Walker Denial    发布于 : 2021-08-12

版本管理规范

1. 提交日志前缀说明

类型 描述
build 修改项目构建系统（例如：glup、webpack、rollup 的配置等）的提交
ci 修改项目继续集成流程（例如：Travis、Jenkins、GitLab CI、Circle 等）的提交
docs 文档更新
feat(feature 简写) 新增功能
merge 分支合并
fix bug 修复
perf 性能, 体验优化
refactor 重构代码（既没有新增功能，也没有修复 bug）
style 不影响程序逻辑的代码修改（修改空白字符，格式缩进，补全缺失的分号等，没有改变代码逻辑）
test 新增测试用例或是更新现有测试
revert 回滚某个更早之前的提交
chore 不属于以上类型的其他类型

2. 分支创建流程

以 feature 为例：

  1. 确认需求及需求版本号；
  2. 从 development 分支克隆出分支，命名规则：feature/{版本号}；
  3. 开发人员拉取 feature/{版本号} 分支代码并创建自己开发分支，命名规则：feature/{版本号}_{开发人员名称} (如果当次开发人员需求多，可以再次细分为多个部分，例如：feature/{版本号}_{开发人员名称}_part1)；
  4. 开发人员开发并自测完毕后，代码提交到 feature/{版本号} 分支；
  5. 整体需求测试通过后，feature/{版本号} 代码合并到 development 分支；
  6. 按照正常发布流程进行发布；

3. 分支保留规则

  1. 保留每次需求汇总分支，示例：feature/{版本号}；
  2. 删除当次开发人员个人分支；
