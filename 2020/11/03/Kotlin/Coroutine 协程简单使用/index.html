Coroutine 协程简单使用

作者 : Walker Denial    发布于 : 2020-11-03

1. 协程是什么？

在使用之前，需要了解什么是 协程，先看一下 协程 的定义：

Coroutines are computer program components that generalize subroutines for non-preemptive multitasking, by allowing multiple entry points for suspending and resuming execution at certain locations. Coroutines are well-suited for implementing more familiar program components such as cooperative tasks, exceptions, event loops, iterators, infinite lists and pipes.

中文翻译如下：

协程（英语：coroutine）是计算机程序的一类组件，推广了 协作式多任务 的 子程序，允许执行被挂起与被恢复。相对子例程而言，协程更为一般和灵活，但在实践中使用没有子例程那样广泛。协程更适合于用来实现彼此熟悉的程序组件，如 协作式多任务、异常处理、事件循环、迭代器、无限列表 和 管道。

协程是一种在程序中处理并发任务的方案；也是这种方案的一个组件。
在协程中，是没有并发、并行概念的。

在 kotlin for Android 中，kotlin 只是一个线程框架，线程所具有的并发、并行等问题，协程也会一个不落。究其原因在于在 Android 开发中 kotlin 需要编译成能被 JVM 识别的字节码，是基于 线程 的，所以不存在协程效率比线程高等情况，协程只是对线程的一个包装，是无法绕开线程的问题的。

2. 协程怎么写？

// 使用 GlobalScope 作用域，通过 launch 开启一段协程
GlobalScope.launch {
    printThreadInfo()
}

private fun printThreadInfo(prefix: String = "") {
    Log.i("CoroutineDemo", "${if(prefix.isNullOrBlank()) "" else "$prefix, "}current thread is ${Thread.currentThread().name}")
}

使用上述代码输出结果如下：

current thread is DefaultDispatcher-worker-1

通过 launch 即可开启一段协程，在 launch 中可以传入 Dispatchers 来指定线程，具体有一下 4 种类型：

  1. Default：默认调度器；
  2. Main：主线程调度器；
  3. Unconfined：未被明确配置的调度器；
  4. IO：流调度器，一般 IO 操作、网络请求 等耗时操作；

3. 串行场景模拟

假设：目前有 3 个网络请求方法，分别是：networkTest1、networkTest2、networkTest3，以及 3 个对应的 UI 更新，分别是：uiTest1、uiTest2、uiTest3。

模拟网络请求以及更新 UI
step 1. 请求接口 1 --> 更新 UI 1
step 2. 请求接口 2 --> 更新 UI 2
step 3. 请求接口 3 --> 更新 UI 3

3.1 普通写法：

// 线程写法
thread {
    networkTest1()
    activity?.runOnUiThread {
        uiTest1()
        thread {
            networkTest2()
            activity?.runOnUiThread {
                uiTest2()
                thread {
                    networkTest3()
                    activity?.runOnUiThread {
                        uiTest3()
                    }
                }
            }
        }
    }
}

输出信息为：

networkTest1(), current thread is Thread-2
uiTest1(), current thread is main
networkTest2(), current thread is Thread-3
uiTest2(), current thread is main
networkTest3(), current thread is Thread-4
uiTest3(), current thread is main

死亡回调有木有？

3.2 RxJava写法：

Observable.just("Test")
            .observeOn(Schedulers.io())
            .doOnNext { networkTestRx1() }
            .observeOn(AndroidSchedulers.mainThread())
            .doOnNext { uiTest1() }
            .observeOn(Schedulers.io())
            .doOnNext { networkTestRx2() }
            .observeOn(AndroidSchedulers.mainThread())
            .doOnNext { uiTest2() }
            .observeOn(Schedulers.io())
            .doOnNext { networkTestRx3() }
            .observeOn(AndroidSchedulers.mainThread())
            .doOnNext { uiTest3() }
            .subscribe()

输出信息如下图所示：

networkTestRx1(), current thread is RxCachedThreadScheduler-1
uiTest1(), current thread is main
networkTestRx2(), current thread is RxCachedThreadScheduler-2
uiTest2(), current thread is main
networkTestRx3(), current thread is RxCachedThreadScheduler-3
uiTest3(), current thread is main

由输出信息可见，RxJava 线程切换没有问题，完美的实现了需求。但是，为了实现线程切换，增加的 线程切换代码量 是 有效执行代码量 的一倍，相比于普通写法，代码量减少了不少，但是线程切换的代码占比较大，那么…还有没有优化空间呢？

3.2 协程 写法：

viewModel.viewModelScope.launch(Dispatchers.Main) {
    networkTests1()
    uiTest1()
    networkTests2()
    uiTest2()
    networkTests3()
    uiTest3()
}

输出信息为：

networkTest1(), current thread is DefaultDispatcher-worker-1
uiTest1(), current thread is main
networkTest2(), current thread is DefaultDispatcher-worker-3
uiTest2(), current thread is main
networkTest3(), current thread is DefaultDispatcher-worker-1
uiTest3(), current thread is main

就这几行看似同步的代码，实现了异步任务、线程切换等问题，由此可见，协程比单纯使用线程方便很多。由日志信息可见，DefaultDispatcher-worker-1 与 DefaultDispatcher-worker-3 并不是同一个线程哟，所以说 协程是单线程的说法是错误的。

4. 并行场景模拟

4.1 普通写法：

thread {
    networkTest1()
    activity?.runOnUiThread { uiTest1() }
}
thread {
    networkTest2()
    activity?.runOnUiThread { uiTest2() }
}
thread {
    networkTest3()
    activity?.runOnUiThread { uiTest3() }
}

直接开启几个线程，更新 UI 时，还要切换回主线程。

4.2 RxJava 写法

Single.zip(
    RetrofitHelper.wanAPI.queryBannerRx(),
    RetrofitHelper.wanAPI.queryChapterRx(),
    { banner, chapter -> "${banner.data?.first()?.title} - ${chapter.data?.first()?.name}" }
).observeOn(AndroidSchedulers.mainThread()).subscribe { combine -> msg.value = combine }

这里实现两个真实并行网络请求，由代码可见，线程能通过 observeOn 自动切换，多个异步也能并行执行，相对于普通写法，代码可读性大大提高，也简洁不少。

4.3 协程写法

viewModelScope.launch(Dispatchers.Main) {
    val banner = async { RetrofitHelper.wanAPI.queryBannerKt().data }
    val chapter = async { RetrofitHelper.wanAPI.queryChapterKt().data }
    msg.value = "${banner.await()?.first()?.title} + ${chapter.await()?.first()?.name}"
}

相比较于 RxJava，代码量更少，在通过 launch 开启线协程后，3 行代码对应 3 个需要的操作，代码更加简洁明了。

5. 总结

协程是什么：协程是一个线程框架。

使用协程的优势：

  1. 耗时代码可以自动分配到后台，任务执行完后也可以自动切回；
  2. 对比 thread，可以避免死亡回调的问题；
  3. 可以用看似同步的代码实现异步任务，使代码更简洁，提高可读性；
